# Drishti — Database Schema

This document describes every table in Drishti's database, why each exists, what shape it takes, what indexes it has, and what RLS policy guards it. It is the source of truth for the schema; Alembic migrations follow this document, not the other way around.

The schema is hybrid by design (see `PRD.md` §3 and `ARCHITECTURE.md` §3.3 for the *why*): a universal raw store plus typed domain tables. Every domain row carries a full provenance chain back to a specific immutable raw API response version.

## 1. Conventions, applied to every table

- **`merchant_id` is the first column** on every table that holds tenant data. It's a `uuid NOT NULL` foreign key to `merchants(id)`.
- **Compound indexes start with `merchant_id`**, always. A query without `merchant_id` is either a bug or an admin path.
- **Primary keys are `uuid`s** (`gen_random_uuid()`) generated by the application layer to enable optimistic local IDs and idempotent retries. `bigserial` IDs would couple us to insertion order and break replayability.
- **Provenance fields**, on every domain table: `source`, `source_record_id`, `raw_record_id` (FK to `source_records`), `sync_run_id` (FK to `sync_runs`), `synced_at`.
- **Timestamps**: `created_at` / `updated_at` are application-managed `timestamptz`. `synced_at` records when the row was last touched by a sync.
- **Soft delete is not used.** If a record disappears from the source, we mark `status='deleted'` on a re-sync but keep the row and its `source_records`. Deletes are extraordinary and audited.
- **All `_at` columns are `timestamptz`**, not `timestamp`. Indian D2C operates across time zones at scale.
- **JSONB columns** use `jsonb_path_ops` GIN indexes when queried; otherwise no index.
- **Money is stored as integer paise** (`bigint`). Never use floats for currency. Currency code stored alongside.
- **RLS is on for every table**. Policy: `USING (merchant_id = current_setting('app.current_merchant_id', true)::uuid)`. Tables that don't carry merchant data (e.g., `merchants` itself) have a different policy documented inline.

### 3.2 `source_records`

The universal raw store. Every API response item lands here as append-only immutable JSONB before normalization runs. This is what makes the citation contract real: every number in every domain row traces back through `raw_record_id` to a literal byte sequence stored here.

| Column | Type | Notes |
|---|---|---|
| `merchant_id` | `uuid FK` | |
| `id` | `uuid PK` | The `raw_record_id` other tables FK against |
| `sync_run_id` | `uuid FK` | The run that fetched this record |
| `source` | `text NOT NULL` | |
| `resource` | `text NOT NULL` | |
| `source_record_id` | `text NOT NULL` | The source's own ID (Shopify order ID, Shiprocket AWB, Razorpay payment ID) |
| `endpoint` | `text NOT NULL` | The literal API path that returned this, e.g. `GET /admin/api/2026-01/orders/5234567890123.json` |
| `fetched_at` | `timestamptz NOT NULL` | |
| `payload` | `jsonb NOT NULL` | The raw response item, byte-for-byte (after sanitization for fixtures only) |
| `payload_hash` | `text NOT NULL` | SHA-256 of payload; used for idempotency |
| `created_at` | `timestamptz NOT NULL` | |

**Indexes:** `(merchant_id, source, source_record_id, payload_hash)` unique, `(merchant_id, source, source_record_id, fetched_at DESC)`, `(merchant_id, fetched_at DESC)`, `(merchant_id, source, resource, fetched_at DESC)`. The unique compound deduplicates identical payload versions; changed source payloads insert a new row and domain rows point to the raw version that produced them.

**Why append-only:** citations must resolve to the exact raw payload version used for the cited normalized value. If the source record changes later, a new `source_records` row is inserted and the normalized domain row's `raw_record_id` advances on upsert; historical citations and audit logs still point at the older raw payload.

## 7. The cross-source join — `order_links`

This is the table that makes Drishti useful. Without it, "which orders are losing money?" can't be answered: order data lives in `orders` (Shopify), shipping cost in `shipments` (Shiprocket), payment status in `payments` (Razorpay). With it, every order has up to one shipment chain and up to one payment chain attached.

| Column | Type | Notes |
|---|---|---|
| `merchant_id` | `uuid FK` | |
| `id` | `uuid PK` | |
| `order_id` | `uuid FK NOT NULL` | → `orders.id` |
| `shipment_id` | `uuid FK NULLABLE` | → `shipments.id`. Nullable because shipments arrive late and may never arrive (cancelled order). |
| `payment_id` | `uuid FK NULLABLE` | → `payments.id`. Nullable because COD orders may not have a Razorpay payment. |
| `linkage_method` | `text NOT NULL` | `order_id_match | metadata_match | manual` |
| `confidence` | `numeric(3,2) NOT NULL DEFAULT 1.00` | Tools that aggregate filter on `confidence >= 0.8` by default |
| `created_at`, `updated_at` | `timestamptz NOT NULL` | |

**Indexes:** `(merchant_id, order_id)` unique, `(merchant_id, shipment_id)` partial, `(merchant_id, payment_id)` partial.

**Linkage logic** (executed during normalization):

1. **Shopify order → Shiprocket shipment.** Shiprocket carries `order_id` field that matches Shopify's order name (e.g., `#1234`) or Shopify's order ID. We try both. `linkage_method = order_id_match`, `confidence = 1.00` if matched.
2. **Shopify order → Razorpay payment.** Razorpay's `notes` or `order_id` field carries the Shopify order ID for prepaid orders. For COD orders the linkage is null and that's expected.
3. **Edge cases.** Orders without shipments yet (recently placed) → row with shipment_id=null, kept current. Orders with multiple shipments (split / RTO + reattempt) → the `order_links` row points at the most recent active shipment. A v1 history table would retain every shipment attempt explicitly.
4. **Unlinked orders are excluded from cross-source duties.** The agent's COD-RTO duty filters `WHERE order_links.shipment_id IS NOT NULL`. Counts of excluded orders are recorded in `agent_runs` — failure mode 3 in `AGENT.md`.

## 10. RLS policies

Every table has RLS enabled. The general policy:

```sql
CREATE POLICY merchant_isolation ON <table>
  USING (merchant_id = current_setting('app.current_merchant_id', true)::uuid)
  WITH CHECK (merchant_id = current_setting('app.current_merchant_id', true)::uuid);
```

Three categorized exceptions:

1. **`merchants`** — uses `id` instead of `merchant_id`:
   ```sql
   USING (id = current_setting('app.current_merchant_id', true)::uuid)
   ```
2. **Tables without merchant_id** (none in v0; if added, must be admin-scope only with a separate `app.is_admin` setting).
3. **Service-role bypasses RLS** by design (Supabase grants the service role `BYPASSRLS`). The application never uses service-role for user-driven requests. Service-role is reserved for migrations and controlled seed/setup paths; isolation validation must run through the same non-bypass application role used by normal requests.

The middleware's contract:

```python
async def merchant_scope_middleware(request, call_next):
    merchant_id = resolve_merchant_id_from_clerk(request)  # raises 403 if not in user's orgs
    async with db.session() as session:
        await session.execute(
            text("SET LOCAL app.current_merchant_id = :mid"),
            {"mid": str(merchant_id)},
        )
        request.state.merchant_id = merchant_id
        request.state.db = session
        return await call_next(request)
```

`SET LOCAL` scopes the setting to the transaction, so cross-request leaks via connection pooling are impossible.

## 11. Migrations

All migrations live in `alembic/versions/`. Conventions:

- One concept per migration (e.g., "add order_line_items," not "shipping refactor").
- Every migration that adds a domain table also adds the RLS enable + policy in the same revision. Schema and security move together.
- Indexes added in a separate revision when the column is added to an existing table (cheaper to roll back).
- The first revision (`0001_initial.sql`) is generated by hand from this document and committed as the baseline.

## 12. What this schema doesn't have, and why

- **No event-sourced domain tables.** Discussed and rejected for v0; replayability is achieved via the append-only raw store + idempotent normalizers, which is enough.
- **No materialized views or rollup tables.** Indexes carry us to the load-harness target; rollups are the next mitigation for larger datasets.
- **No partitioning.** `source_records` will be the first to need it (~10k merchants × ~10k records = 100M rows). The plan: hash partition by `merchant_id` modulo 64, deferred to v1. The current indexes already start with the intended partition key.
- **No vector tables / pgvector.** The chat layer is tool-using over deterministic SQL, not RAG. Adding embeddings would be feature creep against a 7-day v0.
- **No audit-history tables.** `tool_calls`, `sync_runs`, `agent_runs`, and `webhook_deliveries` cover the audit needs we care about. A general `change_log` table is deferred.
